谈谈 iOS 中的 Crash 防护

五一假期的第一天，今天我们来谈谈 iOS 中的 Crash 防护，这两年主要的工作重点都是在这上面，也算是对自己工作的一个总结交代。

前言

工程师们苦 Crash 久矣，尤其是用户感知最为明显的客户端。那居高不下的崩溃率、数不胜数的用户反馈、迟迟无法完成的 KPI，折磨着每一位客户端的开发同学。

加断点再Debug，堆栈瞬间就爆炸，日志输出如雨下，看到异常就害怕；调试一夜没人陪，心想这锅该归谁？回想当初心后悔，不该重构这地雷；翻日志查半天，博客看了千百遍，低头又点一根烟，闪退还是没复现。

Crash 为何如此难以解决，且反复发作？它究竟难在哪里，从客户端工程诞生至今一直困扰着无数优秀的工程师。诚然，确实存在着一些技术瓶颈类的业界难题，但我们工作中遇到的 Crash 问题全都是这种级别吗？

非也，Crash 的本质其实就是 Bug，只要有代码存在，就有产生 Bug 的可能性。同样的，只要你在写代码，就无法完全避免 Crash 问题，既然如此，那我们如何解决 Bug，就如何解决 Crash：

排查 -> 定位 -> 修复

解决 Bug 的流程无外乎就是先排查问题，然后定位具体问题，最后做修复，解决 Crash 亦是如此，总的来说就是：发现问题，找到问题，解决问题，由此可见，二者难点相同。排查和修复都不是最困难的地方，排查可以通过各种上报机制来暴露异常情况，而我们在日常工作中碰到的绝大多数问题点也都是很好处理的，小到你可能仅仅用错了一个修饰符，所以在定位到具体问题后，工程师往往都能在短时间内解决问题。显而易见，修复 Crash 真正的难点就在于如何定位问题。

主动出击

进攻是最好的防守。

在工作中，你不能总是守株待兔，期待 Crash 自己找上门来，主动躺在你的 Todolist 里。你需要在问题大面积爆发前，提前感知，迅速解决，就像碟中谍里的汤姆斯·布鲁克，在悄然无息间拯救地球，深藏功与名。

所以，我们需要一套 Crash 主动防护机制，来帮助工程师们快速处理这些问题。当然，你不可能去指望这套机制帮你写修复代码，所以它只需要能够准确定位到问题点即可。

异常上报

定位问题的前提是能够发现问题，我们需要一个可以帮我们将异常情况上报给工程师来进行排查的机制，它的基本工作原理很简单：1.捕获异常 2.将其上报。

捕获异常

市面上能够找到的统计崩溃等异常的 SDK 非常多，友盟、Bugly、GrowingIO 等等，他们通常都是将捕获到的信息日志上传到对应的服务器上，以此来做一些后续的统计和分析。在一些规模较小的团队当中，无法面面俱到，直接引用这些 SDK 是可以接收的，但如果你的团队对应用性能有一定要求，那么这些 SDK 就会显得有些 “不够用” 了。

作者本人对市面上主流的统计 SDK 都有过较长时间的使用经验，发现无论哪一家 SDK 都或多或少存在这三个问题的其中之一，个别 SDK 甚至都存在：上报延迟 、 异常覆盖率低 以及 日志表述模糊。

总的来说，付费的 SDK 表现的稍好一些（避免广告嫌疑这里就不点名了），但价格昂贵直接拉升了开发成本；而免费的则大多不尽人意，不过人家都让你白嫖了还想奢望什么。

除了 BAT 和同级别的大厂，很少有公司会允许技术团队花费大量时间精力去开发和维护一套完整的上报监控、异常分析的工具流。那么对于一些小而精的团队又该如何去做这件事呢？我们之前说了，市面上主流的统计 SDK 都或多或少存在一些问题，但它们只是存在问题，而不是完全无法使用，通常他们都可以满足 70% 的使用需求，剩下的这 30% 就需要我们来进行 “补充”。

前面提到的三个通病，我们只要针对进行解决，就可以很好的补充这套上报机制。

及时上报

由于我们需要在用户感知之前提前发现问题，所以需要在研发和提测阶段尽可能早的暴露出问题，这样一来可以在崩溃现场或第一时间捕获崩溃信息，然后将崩溃堆栈信息及时反馈给我们的工程师，尤其会大大提高测试和开发之前的沟通效率。
